# Python 3.14 floor; remove markitdown

- **Issue:** TBD (new issue to be filed; supersedes [#3757](https://github.com/nexi-lab/nexus/issues/3757), which was closed by merged [#3789](https://github.com/nexi-lab/nexus/pull/3789))
- **Date:** 2026-04-16
- **Status:** Design approved

## Summary

Raise the Python floor for the Nexus repo to 3.14 across every `pyproject.toml`, every CI workflow, every Dockerfile, and every issue template. Delete `markitdown` from the codebase entirely: remove the core dependency, remove the `nexus[markitdown]` optional extra, delete `markitdown_parser.py` and `markitdown_provider.py`, and purge all import/registration/docstring references across `src/`, `tests/`, and user-facing docs. Replace conditional `pdf-inspector` dep with an unconditional pin. Remove the `compression.zstd` fallback pin (stdlib on 3.14). Bump mypy's `python_version` to 3.14.

PR #3789 has already landed the pdf-inspector integration and the `[markitdown]` extra. This change takes the next step: commits Nexus to Python 3.14 only, with pdf-inspector as the sole local parser.

## Motivation

**Two goals that collapse into one change:**

1. Shed the `markitdown[all]` dependency and its ~200 MB transitive tree (pandas, numpy, lxml, onnxruntime, pdfminer, python-pptx, mammoth, openpyxl, ...). `markitdown`'s Python 3.14 sdist is not compatible with upstream onnxruntime/magika, leaving users on 3.14 with no markitdown path. Keeping it — even as an optional extra — costs maintenance effort and CI cycles for a dependency that can only serve 3.12/3.13 users.
2. Commit to a single Python version. Current develop pins `pdf-inspector; python_version == '3.12'` and `markitdown; python_version == '3.13'`, producing two distinct parse-stack matrices plus a third "neither" branch on 3.14. Bumping the floor to 3.14 deletes the matrix; one version, one parser, one stack.

`pdf-inspector` installs cleanly on 3.14 via sdist + `PYO3_USE_ABI3_FORWARD_COMPATIBILITY=1` + a Rust toolchain. The Docker image already sets the env var and ships Rust; developer installs already have Rust. Bare-metal `pip install` on 3.14 without Rust is the only scenario that breaks — documented as an explicit install requirement.

## Non-goals

- New parsers for Office / HTML / EPUB / audio. Users who need those formats install `markitdown` manually (outside the `nexus` extras) or bring their own `ParseProvider`. Defer per-format providers until a real user request arrives.
- pdf-inspector upstream changes. Shipping cp314 wheels or moving to ABI3-stable is the upstream maintainer's call; not in scope here.
- Keeping 3.12/3.13 support in any form. No dual-matrix, no wheel builds for older interpreters, no compatibility shims.
- Smart OCR routing. `pages_needing_ocr` stays in metadata; no orchestration added.

## Precondition

pdf-inspector installs on Python 3.14. Verified locally:

```
PYO3_USE_ABI3_FORWARD_COMPATIBILITY=1 pip install pdf-inspector
→ Successfully built pdf_inspector-0.1.1-cp314-cp314-macosx_11_0_arm64.whl
```

Docker sets the env var. Dev installs have Rust. Docs must call out the `pip`-without-Rust case.

  - Remove `markitdown[all]>=0.1.0; python_version == '3.13'`.

## Validation

Run locally on Python 3.14 before opening the PR:

1. `python3.14 -m venv .venv && source .venv/bin/activate`.
2. `PYO3_USE_ABI3_FORWARD_COMPATIBILITY=1 uv sync` — expect clean install.
3. `uv run pytest` — full unit suite.
4. `uv run mypy src/` — with the bumped `python_version = "3.14"`.
5. `uv run ruff check .`.
6. Docker smoke test: `docker build` + `docker run` a container that parses a real PDF end-to-end (confirms pdf-inspector works under the image's Rust + env-var setup).

CI on 3.14 reproduces steps 2–5. A green CI run plus the Docker smoke test is the shipping signal.

## Risks and mitigations

- **Bare-metal `pip install nexus` on 3.14 without Rust fails** during pdf-inspector sdist build. Mitigation: explicit doc note in README plus install-instructions page. Severity: medium; users on Docker or dev setups (the primary install paths) are unaffected.
- **Transitive deps without 3.14 wheels.** Preemptively pinned (aiohttp, frozenlist, protobuf, grpcio). Unknowns surface during step 2 of validation. Mitigation: bump the offending dep, or file upstream if no released version supports 3.14. Severity: medium.
- **`pip install nexus[markitdown]` installs break after the bump.** The `[markitdown]` extra no longer exists; `pip` will error with "Requested extra not provided". Mitigation: explicit breaking-change entry in the PR body; changelog note; updated README install table. Severity: low.
- **Documentation drift.** README and user-guide snapshots still reference markitdown. Tracked in the "Docs and templates" section; a final `grep -ri markitdown docs/ README.md` audit guards completion.
- **uv.lock churn.** Full relock may touch unrelated packages. Mitigation: review `uv.lock` diff in the PR, flag unexpected non-markitdown changes for a separate PR if large.

## Rollback

Single PR, single squash merge. Revert via `git revert <squash-sha>` on develop. No schema migrations, no data migrations, no config breaking changes beyond the install surface. If the 3.14 runtime surfaces a blocker post-merge, revert and re-land in two pieces:

1. Markitdown removal at the current 3.12/3.13 floor.
2. Python floor bump with pdf-inspector as the only parser.

Pieces can ship independently if needed.
